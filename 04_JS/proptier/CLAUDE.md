# proptier 프로젝트 규칙

이 프로젝트에서 작업할 때는 항상 아래 규칙을 지킨다.

## 대화 방식

- 불필요한 인사말 없이 바로 답한다.
- 확실하지 않은 사실, 날짜, 수치, 출처는 먼저 불확실하다고 말한다.
- 간단한 질문은 짧게, 복잡한 작업은 충분히 자세히 답한다.

## 변경 통제

- 사용자가 요청하지 않은 파일, 문단, 구조는 바꾸지 않는다.
- 큰 변경, 삭제, 덮어쓰기, 외부 전송은 먼저 무엇을 바꿀지 설명하고 확인을 받는다.
- 작업이 끝나면 변경한 내용과 확인이 필요한 점을 짧게 정리한다.

## 개발 작업 안전

- 현재 요청과 직접 관련된 파일과 코드만 수정한다.
- 기술 스택과 테스트 명령은 프로젝트 규칙을 따른다.

## 기술 스택 / 구조

- HTML/CSS/JavaScript만 사용한다 (다른 언어·프레임워크 도입 금지).
- SPA(단일 페이지 애플리케이션) 구조로 작성한다.
- Provider Pattern을 유지한다 (Mock Provider ↔ Real Provider 교체 가능한 구조).
- UI 레이어는 Storage를 직접 호출하지 않는다 (반드시 Provider를 경유한다).

## 개발 프로세스

- 모든 기능은 Harness(테스트/검증 코드)를 먼저 작성한 뒤 구현한다.
- 하나의 프롬프트(작업 단위)에는 하나의 기능만 구현한다.
- 기존 코드는 삭제하지 않는다 (필요 시 확장/추가로 처리).

## 기억과 연속성

- 중요한 결정은 `MEMORY.md`에 남긴다.
- 세션을 끝낼 때 완료한 일, 진행 중인 일, 다음 할 일을 요약한다.

## 코드 스타일

- 모든 함수에 JSDoc을 작성한다.
- 함수는 40줄 이하로 유지한다.
- Console Error가 발생하지 않도록 한다.
